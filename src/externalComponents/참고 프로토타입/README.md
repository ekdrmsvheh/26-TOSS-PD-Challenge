# 모두의 한 시간 — 참고 프로토타입

> 이 폴더는 `TOSS 프디 챌린지 2026` 리서치 폴더의 최종 확정 프로토타입을 스타터킷으로 옮긴 스냅샷이다.
> 버전 히스토리와 원본은 원본 리서치 폴더에 그대로 남아있다.  
> 기준: [프로젝트 정의](../../../docs/product/프로젝트%20정의/프로젝트%20정의.md), [UX flow](../../../docs/product/UX%20flow/UX%20flow.md), [제품 데이터 정책](../../../docs/product/정책/제품%20데이터%20정책.md)  
> 상태: 전체 주최자 플로우 구현. `convert-external` 스킬로 MUI 컴포넌트 변환 대상

## 목적

이 프로토타입은 C안을 메인 구조로 확정하고, 이전에 정의한 일정 탐색 이후의 주최자 플로우를 다시 반영한 것이다.

핵심은 주최자가 가능한 시간을 찾은 뒤, 그 후보를 바로 확정하는 것이 아니라 `요청 정보 작성 → 참석자 검토 → 응답 현황 → 조정 필요 처리 → AI 조정안 검토 → 주최자 확정` 순서로 불확실성과 조정 비용을 줄여가는 경험을 확인하는 것이다.

## 전체 흐름

```text
미팅 일정 탐색
→ 요청 정보 작성
→ 일정 검토 요청 발송
→ 참석자 응답 현황 확인
→ 미응답 리마인드 또는 응답 기한 이후 처리
→ 조정 필요 응답 처리
→ AI 조정안 검토
→ 최종 후보 재계산
→ 주최자 확정
→ 참석자 공유
```

## 주요 결정

- C안만 남기고 A/B 비교 UI는 제거했다.
- 일정 탐색 화면에서 `시나리오 2: 모두 참석 후보 + 빠른 대안`, `시나리오 3: 필수 공통 시간 없음`을 전환해 테스트할 수 있게 했다.
- 시나리오 2는 추천 후보 3개를 한 번에 참석자에게 검토 요청하는 구조로 변경했다.
- 시나리오 3은 추천 시간이 아니라 현재 조건 안에서 조정 요청 가능한 일정 1개를 보여주고, 바로 초대 작성으로 넘어가지 않게 했다.
- 이번 범위의 조정안은 `필수 참석자의 겹친 일정이 다른 회의에서 선택 참석자인 경우`만 고려한다.
- 미팅 길이, 일정 범위, 필수/선택 변경은 조정안이 아니라 `조건 변경 제안`으로 분리한다.
- 특정 시간 제외 기능은 이번 범위에서 제외했다.
- 라벨은 `미팅 길이`, `일정 범위`, `탐색 시간`으로 정리했다.
- `미팅 길이`, `일정 범위`, `탐색 시간`은 패널에서 값을 누르면 프리셋과 직접 입력을 선택하는 방식으로 설정한다.
- 참여 인원 설정은 좌측 패널에서 길게 노출하지 않고, 주최자/참여자/검색/역할 선택을 포함한 모달에서 진행한다.
- 요청 정보 작성 단계에서 미팅 목적, 응답 기한, 첨부 자료, 참석자별 필요한 이유를 입력한다.
- 참석자는 캘린더를 다시 입력하지 않고 `가능`, `비선호지만 가능`, `조정 필요`, `불가`로 숨은 조건만 응답한다.
- 참석자의 숨은 조건 확인 전에는 확정 버튼을 제공하지 않는다.
- 필수 참석자의 `확인 중`, `미응답`, `조정 필요` 상태는 가능으로 간주하지 않는다.
- `조정 필요`는 확정 동의가 아니며, AI는 조정 요청 가능 근거와 조건 변경 제안을 설명하고 주최자가 최종 판단한다.

## 검증할 것

| 장면 | 확인할 것 |
|---|---|
| 일정 탐색 | 조건을 조정하며 가능한 시간을 다시 찾는 흐름이 자연스러운가 |
| 요청 정보 작성 | 참석자가 응답하기 위해 필요한 맥락이 충분한가 |
| 응답 현황 | 미응답·비선호·조정 필요가 분리되어 이해되는가 |
| 조정 필요 처리 | AI 조정안이 참석자에게 전가되는 비용을 숨기지 않는가 |
| 최종 추천 | 응답과 조정 비용을 반영한 추천 순서가 납득되는가 |
| 확정 공유 | 최종 결정이 자동 추천이 아니라 주최자 판단으로 보이는가 |
