# 공개 근거 자료

최종 확인일: 2026-07-06. 이 문서는 토스의 비공개 채용 정보가 아니라 공개된 1차 자료를 찾아가기 위한 근거 지도다.

## 출처 우선순위

1. 2026 프로덕트 디자이너 챌린지 페이지
2. 현재 토스 프로덕트 디자이너 채용 공고
3. 챌린지 페이지가 직접 연결한 공식 아티클
4. 과거 토스 공식 인터뷰와 디자인 문화 아티클

자료가 충돌하면 더 최신이고 과제와 직접 관련된 출처를 따른다.

## 챌린지 공식 기준

출처: [토스 프로덕트 디자이너 챌린지 2026](https://toss.im/career/designer-challenge-2026)

- 이전 회사에서 한 일과 무관하게 제출한 솔루션만으로 잠재력을 평가한다.
- 과제는 같은 회사 동료 6명이 다음 주까지 1시간 회의를 잡는 경험을 설계하는 것이다. 불가·선호 조건과 필수·선택 참석자 단서가 포함된다.
- 제출 글은 발견한 사용자 문제와 문제인 이유, 해결 방법, 해당 방식으로 설계한 이유에 답해야 한다.
- URL로 실제 사용 흐름을 보여줘야 한다. 문제 정의 없이 결과 화면만 제시하는 것은 과제 취지에 맞지 않는다.
- 기기와 형식은 자유이며 전 과정에서 AI를 사용할 수 있다.
- 공식 평가축은 `문제 정의`, `솔루션 설계`, `시각적 완성도` 세 가지다.

## 2022 챌린지에서 확인되는 보조 근거

출처: [디자이너의, 디자이너에 의한, 디자이너를 위한 채용 설계하기](https://toss.tech/article/2022-product-designer-challenge) — 2022-09-22

- 당시 공식 PASS 기준은 단편적인 UX 오류가 아니라 제품 전체 경험을 근본적으로 해결했는지, 문제 정의·개선안·성과가 논리적으로 연결되는지, 지원자의 뚜렷한 강점이 보이는지였다.
- 한 사람의 취향에 좌우되지 않도록 여러 디자이너가 검토했고 최종 합격 과제는 최대 5명이 확인했다.
- UI 에셋과 템플릿을 제공해 GUI 제작보다 문제 해결에 집중하도록 설계했다.

출처: [경험이 부족한 나도 토스에 지원해도 될까?](https://toss.tech/article/product-designer-challenge-review) — 2022-10-04

- 합격자는 사용자가 제품에서 얻으려는 가치를 먼저 정의하고, 목적 달성을 막는 치명도를 기준으로 문제를 High·Medium·Low로 나눴다.
- 여러 문제를 모두 푸는 대신 가장 중요한 문제와 개선 목표에 집중했다.
- 문제 풀이가 순서대로 전달되도록 한 화면에 여러 내용을 넣지 않고 텍스트와 이미지를 구분했다.

2022 과제는 UT 영상과 기존 UI 에셋을 제공한 개선형 과제이고, 2026 과제는 문제 탐색부터 시각적 완성도까지 지원자가 책임지는 생성형 과제다. 2022의 산출물 형식을 복제하지 말고 `사용자 가치 → 치명도 기준 → 우선순위 → 개선 목표 → 설계`의 사고 구조만 보조 근거로 사용한다.

## 반복되는 제품 설계 신호

출처: [문제 원인의 원인을 찾아서](https://toss.tech/article/real-reason) — 2023-03-02

- 표면적인 불편에 멈추지 않고 “왜”를 반복해 불필요한 입력 자체를 없앨 수 있는 근본 원인을 찾았다.

출처: [내 아이디어를 너무 믿지 마세요](https://toss.tech/article/dont-trust-your-gut) — 2023-03-28

- 디자이너가 확신한 음성 이력서 아이디어가 UT에서 더 부담스러운 경험으로 드러나자 가설과 방향을 바꿨다.
- 프로토타입과 UT는 아이디어를 입증하는 의식이 아니라 틀릴 가능성을 빠르게 확인하는 수단으로 쓰였다.

출처: [추천할 때는 제일 좋은 것 하나면 된다](https://toss.tech/article/21004) — 2023-03-23

- 사용자의 선택 부담을 줄이는 하나의 추천이 여러 선택지를 보여주는 안보다 높은 전환을 만들었다.
- 저자는 실험 변수가 완전히 통제되지 않았다는 한계도 공개했다.

허용되는 추론: 문제의 양보다 선택 기준, 아이디어의 독창성보다 반증 가능성, 결과 수치만큼 실험 한계에 대한 정직함을 평가한다.

## 현재 직무가 요구하는 역량

출처: [토스 Product Designer 채용 공고](https://toss.im/career/job-detail?company=%ED%86%A0%EC%8A%A4&job_id=4326315003&sub_position_id=4326315003)

- 프로덕트 디자이너는 6~8명 규모 팀의 1인 디자이너로 사용자 경험에 대한 최종 책임과 권한을 가진다.
- 비즈니스, 정책, 운영까지 고민하며 제품이 고객에게 닿는 순간까지 책임진다.
- 정성·정량 근거로 근본 문제를 정의하고, 명확한 가설을 담은 화면을 제안하며, VOC 수집부터 프로토타입까지 주도하는 역량을 요구한다.
- 복잡한 규칙을 쉬운 경험으로 풀고, 앱·웹 UI를 픽셀 단위로 완성하며, 다양한 직군을 설득하고 조율하는 역량을 요구한다.
- 포트폴리오에서는 안정적이고 직관적인 화면, 논리적이고 사용자 중심적인 문제 해결, 목표와 사용자 최적안의 균형을 중점 평가한다.

## 직무 경계 변화에서 읽을 수 있는 신호

출처: [토스가 디자인 직무를 2개로 줄인 이유](https://toss.tech/article/Designer) — 2026-04-03

- Product Designer와 Tools Product Designer가 통합되면서 모바일·PC 구분이 직무의 기준에서 사라졌다.
- 화면 종류나 도구보다 사용자의 맥락과 문제를 파악하고 해결하는 역할로 프로덕트 디자인을 정의한다.
- 도구와 AI가 하드스킬 습득 비용을 낮출수록 무엇이 좋은 경험인지 판단하는 능력을 중요하게 본다.

허용되는 추론: 판단 범위의 넓이와 필요할 때 도구 경계를 넘는 태도를 살핀다. 모든 지원자가 코딩해야 한다고 추론하지 않는다.

## 프로토타입과 AI 활용에서 읽을 수 있는 신호

출처: [디자이너가 시안 대신 앱을 만든 이유](https://toss.tech/article/deadend) — 2026-06-16

- 인터랙션이 중요한 프로젝트에서 정지 시안에 머물지 않고 SwiftUI 프로토타입을 만들고 반복 검증했다.
- AI는 구현을 도왔지만 경험 설명, 대안 선택, 실기기 결과 판단은 디자이너가 담당했다.
- 동작하는 결과물이 개발자와의 모호함을 줄이고 실행 가능한 스펙 역할을 했다.

허용되는 추론: 핵심 경험을 판단할 수 있게 만드는 프로토타입과 사람의 판단권을 유지하는 AI 활용을 긍정적 신호로 본다. 코드 작성 자체에는 점수를 주지 않는다.

출처: [AI로 바꾼 제품 설계의 순서](https://toss.tech/article/chatbot) — 2026-06-17

- 수치로 확인된 퍼널 문제와 익명화된 실제 상담 데이터에서 출발했다.
- 시나리오와 프로토타입을 함께 발전시키고 현실적인 분기를 일찍 검증했으며 반복 실패를 공통 규칙으로 바꿨다.
- 목표 경험을 먼저 설계한 뒤 필요한 데이터, API, 운영 도구를 역산했다.

허용되는 추론: 현실적인 사례, 빠른 검증, 재사용 가능한 규칙, 의사결정을 바꾸는 근거를 살핀다. 이 사례의 프로세스를 모든 과제에 강제하지 않는다.

## 협업·시스템·의사결정에서 읽을 수 있는 신호

출처: [서비스 개편, 토스팀 디자이너 4명에게 직접 들어봤습니다](https://toss.im/tossfeed/article/toss-designer-interview) — 2019-03-11

- 역할을 화면뿐 아니라 정보 설계, UX 라이팅, 요구사항 정립, 설득, 피드백 수렴까지 포함해 설명했다.
- 정량 지표와 정성 근거에 설득될 때 팀의 의사결정이 빠르게 이뤄진다고 설명했다.
- TDS를 일관된 경험과 반복 작업 감소를 위한 공통 디자인 언어로 소개했다.

오래된 글이므로 정확한 조직 구조보다 근거 기반 의사결정과 시스템 차원의 효율화 신호를 참고한다.

출처: [토스는 왜 디자인 컨퍼런스를 여는 걸까?](https://toss.im/tossfeed/article/toss-design-conference-simplicity21) — 2021-08-19

- 반복 실험, 사용자 인터뷰, 좋은 수준을 넘어선 경험, 다른 관점의 문제 재정의, 서비스 간 일관성, 학습 공유를 강조했다.

출처: [디자인 컨퍼런스 심플리시티24](https://toss.im/tossfeed/article/simplicity24) — 2024-11-12

- 문제의 본질을 파고들어 모호함을 임팩트로 바꾸기, 복잡한 이해관계에서 솔루션 찾기, 역할과 제품의 틀을 넘기 등을 주제로 제시했다.

허용되는 추론: 근본 원인, 이해관계자의 선택과 대가, 시스템 확장성, 학습, 임팩트를 질문한다. 이를 추가 공식 평가축으로 만들지 않는다.

## 면접관 역할의 제한

- 챌린지 페이지나 현재 채용 공고에 직접 적힌 내용만 `공식 기준`이라고 말한다.
- 아티클에서 도출한 결론은 `공개 사례 기반 추론`이라고 말한다.
- 지원자가 증거를 제공하지 않은 주장은 `검증 필요`라고 말한다.
- 실제 면접관, 비공개 가중치, 합격선, 선호 솔루션을 안다고 주장하지 않는다.
- 토스와 비슷한 시각 스타일을 제품 디자인 역량과 동일시하지 않는다.
- 기능 수, 리서치 양, AI 기술의 화려함을 판단력으로 착각하지 않는다.
- 시각적 완성도로 빠진 흐름을 덮지 않고, 논리가 좋다는 이유로 약한 UI를 용인하지 않는다.
