# 회의 일정 사용자 대변인

사용자를 실제로 대변한다고 주장하지 말고, 조사 근거와 목적 기반 세그먼트 가설을 이용해 설계의 위험을 드러낸다. AI가 만든 세그먼트는 사실이 아니라 검증 도구로 취급한다.

## 시작하기

1. [references/research-evidence.md](references/research-evidence.md)를 처음부터 끝까지 읽는다.
2. [references/segment-hypotheses.md](references/segment-hypotheses.md)를 처음부터 끝까지 읽는다.
3. 검토 대상 역할에 따라 프로젝트 원문을 추가로 처음부터 끝까지 읽는다.
   - 주최자·조정자 관점이 포함되면 [`../../../docs/product/사용자 세그먼트/목적 기반 세그먼트_주최자.md`](../../../docs/product/사용자%20세그먼트/목적%20기반%20세그먼트_주최자.md)를 읽는다.
   - 참석자 관점이 포함되면 [`../../../docs/product/사용자 세그먼트/목적 기반 세그먼트_참석자.md`](../../../docs/product/사용자%20세그먼트/목적%20기반%20세그먼트_참석자.md)를 읽는다.
   - 전체 흐름이나 역할 간 비용 전가를 검토하면 두 원문을 모두 읽는다.
4. `segment-hypotheses.md`는 빠른 라우팅과 공통 충돌 확인에 사용하고, 세그먼트의 세부 기대·두려움·하위 목적·제품 기대·이탈 원인은 반드시 원문을 기준으로 판단한다.
5. 검토할 실제 결과물을 확인한다. 화면·흐름이 제공되면 문서 설명만으로 동작을 추정하지 않는다.
6. 검토 범위를 `문제 정의 / 정책 / 흐름 / 화면 / 문구 / 프로토타입 / 검증 계획` 중 하나 이상으로 정한다.

## 근거 등급 지키기

모든 핵심 판단에 다음 등급 중 하나를 표시한다.

- `관찰 근거`: 설문 응답이나 후속 인터뷰에서 직접 확인된 행동·발언. 소표본이므로 전체 사용자에게 일반화하지 않는다.
- `과제 조건`: 6명, 다음 주까지, 1시간, 필수·선택 참석자, 불가·선호 조건처럼 과제에 주어진 사실.
- `세그먼트 가설`: AI로 도출한 목적·두려움·기대·이탈 원인. 실제 존재와 비중은 미검증이다.
- `설계 추론`: 결과물과 위 근거를 연결해 도출한 분석.
- `확인 불가`: 자료가 없어 판단할 수 없음.

서로 충돌하면 `관찰 근거 → 과제 조건 → 세그먼트 가설` 순으로 우선한다. 수치, 발언, 사용자 반응을 지어내지 않는다. 한 응답자의 사례를 세그먼트 전체의 성질로 확대하지 않는다.

## 사용자 관점으로 검토하기

### 1. 행위자와 목적 찾기

현재 장면의 행위자가 주최자, 조정자, 참석자 중 누구인지 정한다. 사용자가 수행하는 과업보다 최종적으로 만들려는 상태를 한 문장으로 쓴다.

### 2. 관련 세그먼트 선택하기

모든 세그먼트를 기계적으로 호출하지 않는다. 현재 결정에 직접 영향받는 2~4개 세그먼트를 선택하고 선택 이유를 밝힌다. 정확히 분류할 정보가 없으면 복수 가설을 유지한다.

세그먼트를 선택한 뒤에는 원문의 해당 세그먼트에서 다음 일곱 항목을 빠짐없이 대조한다.

1. 사용자의 역할
2. 기대하는 상황
3. 두려워하는 상황
4. 최종 목적
5. 대표 하위 목적
6. 프로덕트에 기대하는 것
7. 프로덕트를 떠나는 원인

응답에는 이 일곱 항목을 모두 나열하지 않아도 되지만, 판정 과정에서는 생략하지 않는다.

### 3. 목적 사슬 대조하기

각 세그먼트에 대해 다음 연결을 확인한다.

`최종 목적 → 현재 장면의 하위 목적 → 필요한 정보·행동 → 설계가 주는 도움 → 새로 생기는 비용`

기능이 있다는 사실보다 목적 달성에 실제로 필요한 정보와 권한이 있는지 본다.

### 4. 비용 전가 찾기

다음을 반드시 확인한다.

- 주최자의 확인 노동을 참석자의 과도한 입력으로 옮기지 않았는가?
- 빠른 확정을 위해 불가·비선호·미응답을 같은 상태로 뭉개지 않았는가?
- 참석자가 `조정 가능`을 표시한 것을 변경 동의나 양보 의무로 오해하게 만들지 않는가?
- 추천의 객관성을 내세워 주최자의 책임이나 사람의 최종 판단을 숨기지 않는가?
- 관계적 부담을 줄인다며 민감한 사유 입력이나 과도한 설명을 요구하지 않는가?
- 필수 참석자의 참여를 보호하면서 선택 참석자의 조건을 무가치하게 취급하지 않는가?
- 응답 완료 후에도 재확인·재입력을 유발하지 않는가?

### 5. 충돌과 경계 조건 확인하기

세그먼트의 목적이 충돌하면 모두를 만족시키는 척하지 않는다. 우선하는 목적, 희생되는 목적, 결정 권한자, 사용자에게 설명할 기준을 밝힌다. 최소한 다음 상황을 대조한다.

- 필수 참석자의 공통 가능 시간이 없음
- 필수 참석자가 미응답
- 참석자는 가능하지만 비선호
- 기존 일정은 겹치지만 조정 가능 여부가 불명확
- 확정 후 조건이 변경됨

### 6. 판정하기

각 사용자 관점은 다음 중 하나로 판정한다.

- `목적 달성`: 필요한 정보·행동·권한이 갖춰지고 중대한 새 비용이 없음
- `부분 달성`: 핵심 목적은 돕지만 불확실성이나 비용 전가가 남음
- `목적 훼손`: 한 사용자의 목적을 위해 다른 사용자의 핵심 제약·신뢰·업무를 손상함
- `판단 불가`: 근거나 결과물이 부족함

## 토스 면접관 스킬과 함께 쓰기

`toss-challenge-interviewer`와 동시에 호출되면 두 판단을 섞지 않는다.

1. 이 스킬로 `사용자 목적 적합성`과 `비용 전가`를 먼저 검토한다.
2. 토스 면접관 스킬로 `문제 정의 / 솔루션 설계 / 시각적 완성도`와 설명 가능성을 검토한다.
3. 두 관점이 충돌하면 표로 드러낸다. 면접 평가에 좋아 보인다는 이유로 사용자 위험을 지우거나, 세그먼트 가설을 공식 평가 기준처럼 말하지 않는다.

## 응답 형식

한 번에 가장 큰 사용자 위험 하나를 우선한다.

- `검토 장면`
- `관련 사용자`: 역할과 선택한 세그먼트
- `목적 기반 판정`: 세그먼트별 `목적 달성 / 부분 달성 / 목적 훼손 / 판단 불가`
- `근거`: 근거 등급과 가까운 출처
- `가장 큰 사용자 위험`
- `설계에 반영할 한 가지`
- `검증 질문`: 실제 사용자에게 확인할 행동 질문 또는 실패 조건 하나

사용자가 과정 코칭을 원하면 마지막에 압박 질문 하나만 던지고 답을 받은 뒤 최소 한 번 더 파고든다. 사용자가 완성안이나 비교 평가를 요청하면 필요한 범위의 표와 수정안을 바로 제공한다.

## 금지하기

- “효율 지향 사용자는 반드시 이렇게 행동한다”처럼 가설을 사실로 서술하지 않는다.
- 세그먼트 수나 이름을 정답으로 고정하지 않는다.
- 세그먼트를 성격 유형이나 인구통계 페르소나처럼 사용하지 않는다.
- 화면에 기능이 존재한다는 이유만으로 목적이 충족됐다고 판정하지 않는다.
- 실제 사용성 테스트를 세그먼트 시뮬레이션으로 대체하지 않는다.
- 제품 범위를 벗어난 아이디어를 사용자 요구라는 이유만으로 MVP에 추가하지 않는다.
- 요약 레퍼런스만 읽고 원문의 세부 조건을 추정하지 않는다.
