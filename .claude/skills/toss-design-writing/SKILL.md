# 토스 디자인 글쓰기

토스 디자인 챕터 블로그에서 관찰한 문장 구조와 서사 방식을 참고해, 디자인 과제 답변을 사람의 판단과 고민이 보이는 글로 바꾼다. 토스 내부 문체를 안다고 주장하지 말고, 공개 블로그 글에서 관찰한 패턴을 응용한다.

## 시작하기

1. [references/toss-blog-style-analysis.md](references/toss-blog-style-analysis.md)를 처음부터 끝까지 읽는다.
2. 사용자가 준 답변이나 초안을 다음 네 가지로 분해한다.
   - `상황`: 어떤 맥락에서 시작했는가
   - `마찰`: 사용자가 실제로 겪는 불편이나 설계자가 발견한 이상함은 무엇인가
   - `질문 전환`: 처음의 문제를 어떻게 다시 물었는가
   - `판단`: 어떤 근거로 무엇을 선택했고 무엇을 버렸는가
3. 부족한 근거를 지어내지 않는다. 수치, 인터뷰 발언, 테스트 결과가 없으면 `정성 관찰`, `과제 조건`, `설계 가설`, `검증 필요`로 구분한다.
4. 답변을 바로 꾸미지 말고, 먼저 가장 약한 한 지점을 찾는다. 보통은 `구체적 장면 없음`, `사용자 마찰 없음`, `선택의 이유 없음`, `결과가 기능 나열로 끝남` 중 하나다.

## 재작성 원칙

### 1. 큰 주장보다 작은 장면으로 시작하기

추상적인 첫 문장을 피한다.

- 약한 시작: `저는 사용자 중심으로 문제를 정의했습니다.`
- 바꿀 방향: `일정을 정하는 과정에서 가장 먼저 보인 건, 시간이 없다는 사실보다 각자가 가진 조건을 서로 설명해야 하는 부담이었어요.`

첫 문단은 `상황 → 이상함 → 질문` 순서로 만든다.

### 2. 문제를 한 번 다시 묻기

토스식 설득은 해결책을 바로 말하지 않고 질문을 바꾼다.

- `정말 필요한 건 가능한 시간을 많이 모으는 걸까?`
- `주최자가 하고 싶은 일은 시간을 고르는 게 아니라, 모두가 납득할 수 있는 결정을 빠르게 만드는 것 아닐까?`
- `참석자가 입력해야 하는 건 모든 일정이 아니라, 절대 안 되는 시간과 조정 가능한 조건 아닐까?`

답변마다 이런 전환 질문을 1개 이상 만든다.

### 3. 기능을 결과가 아니라 판단의 산물로 쓰기

기능 목록은 마지막에만 둔다. 각 기능은 반드시 이 연결로 설명한다.

`관찰한 마찰 → 다시 정의한 문제 → 설계 기준 → 선택한 기능 → 기대되는 행동 변화`

이 연결이 없는 기능은 과제 답변에서 빼거나 한 문장으로 낮춘다.

### 4. 디자인 근거를 사람의 언어로 풀기

전문 용어를 먼저 쓰지 않는다. 사용자가 실제로 알고 있는 말, 행동, 상황으로 설명한 뒤 필요할 때만 용어를 붙인다.

- `정보 구조를 개선했습니다`보다 `사용자가 카테고리를 맞히지 않아도 되게 했습니다`
- `인지 부하를 낮췄습니다`보다 `참석자가 고민해야 하는 선택지를 먼저 줄였습니다`
- `효율성을 높였습니다`보다 `주최자가 다시 묻고 비교하는 일을 줄였습니다`

### 5. 결과보다 달라진 판단을 남기기

마무리는 `무엇을 만들었다`가 아니라 `이 프로젝트를 통해 무엇을 다르게 보게 됐는가`로 끝낸다.

- `그래서 저는 회의 일정 조율을 시간 선택 문제가 아니라, 각자의 조건을 안전하게 드러내고 납득 가능한 결정을 만드는 문제로 정의했습니다.`
- `AI는 결정을 대신하는 장치가 아니라, 사람이 놓치기 쉬운 충돌과 대안을 빠르게 보이게 하는 장치로 두었습니다.`

## 답변 만들기 절차

1. 원문을 한 줄로 요약한다.
2. AI처럼 보이는 문장을 표시한다.
   - 과도한 명사형: `사용자 중심`, `효율적`, `최적화`, `경험 개선`, `문제 해결`
   - 근거 없는 단정: `명확히`, `반드시`, `효과적으로`, `최적의`
   - 포트폴리오식 공허한 결론: `사용자의 편의성을 높였습니다`
3. 답변을 4문단으로 재구성한다.
   - `발견`: 처음엔 당연해 보였지만 다시 보니 이상했던 점
   - `정의`: 문제를 다시 물은 과정
   - `해결`: 선택한 설계와 버린 대안
   - `의미`: 사용자, 운영, 비즈니스, 협업 관점에서 달라지는 점
4. 문장 톤을 다듬는다.
   - 기본 종결은 `-했어요`보다 과제 제출에 맞게 `-했습니다`를 사용하되, 너무 딱딱하면 일부 문장에 `-였죠`, `-라고 봤습니다`, `-에 가까웠습니다`를 섞는다.
   - 한 문단에 긴 문장 1개, 짧은 문장 1~2개를 섞는다.
   - 질문형 문장은 남발하지 말고 전환점에만 쓴다.
5. 마지막에 면접관 관점으로 한 번 더 점검한다.
   - 내 역할이 보이는가?
   - 근거 수준이 정직한가?
   - 기능보다 판단이 먼저 보이는가?
   - 사용자 가치와 제품 임팩트가 연결되는가?

## 응답 형식

사용자가 초안을 고쳐 달라고 하면 다음 순서로 답한다.

- `핵심 진단`: AI처럼 보이는 가장 큰 이유 1개
- `구조 변경`: 원문의 논리를 어떻게 재배열했는지
- `개선 답변`: 바로 붙여 넣어 다듬을 수 있는 문장
- `더 날카롭게 만들 질문`: 답변자가 스스로 보강할 질문 1개

사용자가 “분석부터” 요청하면 reference의 분석을 바탕으로 표를 사용해 설명한 뒤, 스킬을 적용한다.

## 금지하기

- 토스 블로그 문장을 길게 베끼지 않는다.
- 공개 글에서 확인되지 않은 토스 내부 평가 기준이나 문체 원칙을 안다고 말하지 않는다.
- 감성적인 문장만 늘려서 근거 부족을 가리지 않는다.
- 수치나 사용자 반응을 만들어내지 않는다.
- `사용자 중심`, `직관적`, `효율적`, `최적화` 같은 말을 근거 없이 결론으로 쓰지 않는다.
