# 토스 디자인 챕터 블로그 글쓰기 분석

이 문서는 2026년 7월 기준 공개된 토스 테크 디자인 카테고리 글을 관찰해, 과제 답변과 포트폴리오 문장에 응용할 수 있는 글쓰기 패턴을 정리한 것이다. 토스 내부 가이드가 아니라 공개 글 기반 분석이다.

분석 범위:

- 토스 테크 디자인 카테고리 API 기준 전체 136개 글의 제목, 부제, 카테고리, 본문 텍스트를 확인했다.
- 카테고리 분포는 Product Design 69개, Visual Design 32개, UX Research 28개, UX Writing 10개, 기타 Simplicity/접근성/피플/Industrial Design 등이다.
- 아래 5개 글은 최신 AI/제품 설계 흐름을 대표해 문단 구조와 문장 전환을 더 자세히 읽었다.

- `디자이너에게 AI로 뭐든 만들어보라고 한다면`, 토스 디자인 챕터, 2026-06-19, https://toss.tech/article/ai_contest
- `매일 하던 업무를 디자인하기`, 김혜미, 2026-06-17, https://toss.tech/article/todolist
- `AI로 바꾼 제품 설계의 순서`, 김유라, 2026-06-17, https://toss.tech/article/chatbot
- `디자이너가 시안 대신 앱을 만든 이유`, 김민지, 2026-06-16, https://toss.tech/article/deadend
- `하마터면 못생겨질 뻔했다 - 토스 프론트 2 제작기`, 류관준, 2026-04-09, https://toss.tech/article/toss_front

전체 글에서 반복적으로 관찰된 키워드는 `사용자`, `문제`, `그래서`, `과정`, `리서치`, `AI`, `시스템`, `효율`, `기준`이었다. 제목이나 부제에 질문형을 쓰는 글도 136개 중 67개로 많다. 즉 토스 디자인 글은 정답을 선포하기보다 질문을 세우고, 그 질문이 설계 기준으로 바뀌는 과정을 보여주는 경향이 강하다.

## 1. 글의 기본 골격

토스 디자인 글은 대체로 아래 순서로 설득한다.

1. `일상적이거나 업무적인 장면`을 먼저 보여준다.
2. 그 장면에서 당연하게 여겼던 불편을 드러낸다.
3. 문제를 해결책 중심이 아니라 질문 중심으로 다시 정의한다.
4. 처음 시도와 실패, 제약, 버린 대안을 말한다.
5. 해결책이 왜 그 모양이 되었는지 판단 기준을 밝힌다.
6. 결과를 기능 성과보다 `일하는 방식`, `검증 방식`, `사용자 경험의 기준` 변화로 마무리한다.
7. 독자가 적용할 수 있는 질문으로 끝낸다.

과제 답변에 적용하면 `저는 이런 기능을 만들었습니다`보다 `처음에는 A라고 생각했지만, 실제 문제는 B에 가까웠습니다`가 더 설득력 있다.

## 1-1. 글 유형별 차이

전체 디자인 카테고리는 하나의 문체로만 쓰이지 않는다. 과제 답변을 만들 때는 아래 유형 중 어느 쪽에 가까운지 먼저 고른다.

### Product Design형: 문제를 다시 정의하고 선택을 설명한다

가장 과제 답변에 바로 쓰기 좋은 유형이다. 표면 문제, 사용자 행동, 제품 제약, 실험 또는 출시 결과가 연결된다.

자주 보이는 흐름:

1. `처음에는 A가 문제처럼 보였다`
2. `하지만 사용자 흐름을 따라가 보니 B가 더 컸다`
3. `그래서 기준을 C로 바꿨다`
4. `D 기능을 만들었고, E 대안은 이런 이유로 버렸다`
5. `결과적으로 사용자는 F를 덜 하게 됐다`

과제 답변에서는 일정 조율을 `시간 추천 기능`으로 설명하지 말고, `주최자와 참석자가 서로의 조건을 납득 가능한 형태로 맞추는 문제`로 다시 정의해야 한다.

### UX Research형: 현장과 데이터가 추상적 판단을 바꾼다

리서치 글은 대체로 질문으로 시작한다. 사용자를 만나기 전의 가정, 실제 현장에서 깨진 부분, 리서치 방식의 한계, 그 후 설계나 조직 방식이 어떻게 바뀌었는지를 보여준다.

자주 보이는 흐름:

1. `이 문제를 알려면 어떤 사용자를 어디에서 만나야 할까?`
2. `막상 만나보니 우리가 상상한 장면과 달랐다`
3. `사용자의 언어와 제품/조직의 언어가 달랐다`
4. `그래서 리서치 결과를 인사이트에서 끝내지 않고 도구, 기준, 프로세스로 바꿨다`

과제 답변에서는 설문이나 인터뷰를 했다면 `몇 명을 만났다`보다 `그 근거 때문에 어떤 설계 판단을 바꿨는가`를 먼저 말한다.

### UX Writing형: 좋은 문장은 원칙과 시스템으로 확장된다

UX Writing 글은 문장 하나의 감각보다, 여러 사람이 일관되게 잘 쓰도록 만드는 원칙과 시스템을 강조한다. 클릭률, 에러 메시지, 보이스톤 같은 주제를 다루더라도 핵심은 `한 명의 감`을 `팀이 반복 가능한 기준`으로 바꾸는 과정이다.

자주 보이는 흐름:

1. `좋은 문구를 쓰는 문제가 반복됐다`
2. `개별 피드백으로는 한계가 있었다`
3. `그래서 원칙, 예시, 시스템으로 바꿨다`
4. `좋은 문장은 사용자가 다음 행동을 할 수 있게 돕는다`

과제 답변의 문구 설계에서는 `친절하게 안내했습니다`보다 `사용자가 지금 무엇을 선택해야 하는지, 그 선택의 결과가 무엇인지 알 수 있게 문장을 나눴습니다`처럼 쓴다.

### Visual/Brand/Interaction형: 감각을 기준 언어로 번역한다

Visual Design 글은 아름답다는 말로 끝나지 않는다. 감각적 결과물을 만들더라도 그것이 어떤 인상, 사용성, 브랜드 태도, 재사용성, 시스템 확장성에 기여하는지 설명한다.

자주 보이는 흐름:

1. `시각적으로 만들 대상이 있었다`
2. `하지만 예쁘기만 해서는 부족했다`
3. `이 대상이 전달해야 하는 인상과 쓰임을 기준으로 세웠다`
4. `여러 시도를 비교하며 기준에 맞지 않는 안을 버렸다`
5. `결과물이 시스템이나 반복 사용으로 확장됐다`

과제 답변에서 UI 완성도를 말할 때는 `깔끔하게 디자인했습니다`보다 `필수 참석자 조건, 선호 조건, 미응답 상태가 같은 표면으로 보이지 않도록 정보 위계를 분리했습니다`처럼 기준을 붙인다.

### 접근성형: 시스템 문법을 사용자 감각으로 다시 번역한다

접근성 업무일지 글은 시각 정보, 읽는 순서, 역할 정보처럼 제품 내부에서는 당연한 문법을 실제 사용자의 감각으로 다시 검토한다.

자주 보이는 흐름:

1. `제품은 정보를 보여주고 있었지만, 특정 사용자에게는 전달되지 않았다`
2. `시스템의 기본 순서나 표현이 사용자의 실제 이해 순서와 어긋났다`
3. `그래서 보이는 정보, 들리는 정보, 조작 가능한 정보를 다시 배열했다`

과제 답변에서는 접근성을 별도 체크리스트처럼 붙이지 말고, `상태를 색만으로 구분하지 않는다`, `추천 이유를 텍스트로도 설명한다`, `미응답/불가/비선호를 명확히 구분한다`처럼 설계 본문에 녹인다.

## 2. 문제 발견 방식

### 사소한 불편을 무시하지 않는다

`매일 하던 업무를 디자인하기`는 할 일을 옮겨 적는 반복 행위를 개인 습관이 아니라 디자인할 문제로 본다. 핵심은 거대한 리서치보다 `불편한 줄도 모르던 반복`을 발견하는 태도다.

과제 답변 적용:

- 일정 조율에서 `가능 시간 입력` 자체보다, 입력 이후 주최자가 비교하고 다시 묻는 반복을 문제로 볼 수 있다.
- 참석자 입장에서는 `시간 선택`보다 `내 조건을 어디까지 드러내야 하는지 모르는 부담`이 문제일 수 있다.

### 사용자 언어와 시스템 언어의 차이를 포착한다

`AI로 바꾼 제품 설계의 순서`는 사용자는 자기 문제는 알지만 서비스의 카테고리는 모른다는 점을 문제로 삼는다. 이 구조는 일정 조율에도 그대로 쓸 수 있다.

과제 답변 적용:

- 사용자는 `화요일 2시 가능`보다 `그 시간은 회의는 가능하지만 이동 직후라 부담스럽다`처럼 조건을 갖고 있다.
- 제품은 이를 `가능/불가능`으로 단순화하기 쉽다.
- 좋은 설계는 사용자의 조건을 제품이 판단 가능한 형태로 바꾸되, 사용자가 억지로 시스템 언어를 맞히게 하지 않는다.

### 목표를 특정 화면 하나가 아니라 공통 장치로 넓힌다

`디자이너가 시안 대신 앱을 만든 이유`는 데드엔드 화면 하나가 아니라, 앱 안 어디서든 다음 경험으로 이어지는 공통 장치를 목표로 둔다.

과제 답변 적용:

- `후보 시간 추천 화면` 하나가 아니라, 초대부터 확정까지 반복되는 판단 규칙을 설계했다고 말하는 편이 강하다.
- 하나의 화면 개선보다 `불가/선호/조정 가능`을 다루는 시스템 규칙이 더 포트폴리오 신호가 된다.

## 3. 근거 표현 방식

### 숫자는 배경을 짧고 단단하게 만든다

토스 글은 수치를 길게 해석하기보다 문제의 규모를 빠르게 세운다. 예를 들어 고객센터 방문량, 상담 시도 수, 이탈률처럼 맥락을 압축하는 숫자를 먼저 제시한 뒤 구조 문제로 연결한다.

과제 답변 적용:

- 실제 수치가 없으면 만들지 않는다.
- 과제 조건 자체를 숫자 근거로 쓴다: `6명`, `다음 주까지`, `1시간`, `필수/선택 참석자`, `불가/선호 조건`.
- 설문이나 인터뷰가 있으면 대표성보다 `어떤 설계 판단을 바꿨는지`에 집중한다.

### 판단 근거는 비교로 드러낸다

`토스 프론트 2 제작기`는 각 시도마다 좋아 보인 점과 버린 이유를 함께 쓴다. 핵심은 실패를 실패담으로 쓰는 것이 아니라, 기준을 선명하게 만드는 장치로 쓰는 것이다.

과제 답변 적용:

- `캘린더 전체 공유`를 버렸다면 사생활 부담이나 입력 비용을 이유로 말한다.
- `자동 확정`을 버렸다면 납득 가능성과 책임 소재를 이유로 말한다.
- `가능/불가능만 입력`을 버렸다면 참석자의 선호와 조정 가능성을 반영하지 못한다는 이유를 말한다.

## 4. 문장 구조

### 짧은 문장으로 장면을 고정한다

토스 글은 긴 개념 설명 사이에 짧은 문장을 넣어 독자가 문제를 잡게 한다.

응용 패턴:

- `문제는 그 다음이었습니다.`
- `여기서 질문을 바꿔봤습니다.`
- `처음부터 잘 되지는 않았습니다.`
- `이 선택은 편했지만 충분하지 않았습니다.`
- `그래서 기준을 다시 세웠습니다.`

### 질문형 문장이 전환점 역할을 한다

질문은 도입부 장식이 아니라 사고가 바뀐 순간에 쓴다.

응용 패턴:

- `정말 주최자가 필요한 건 가장 빠른 시간 추천일까?`
- `참석자는 모든 가능한 시간을 입력하고 싶은 걸까, 아니면 피해야 할 조건을 안전하게 알리고 싶은 걸까?`
- `AI가 대신 결정하는 것이 좋은 경험일까, 아니면 판단 근거를 더 빨리 보이게 하는 것이 좋은 경험일까?`

### 전문 용어보다 사용자 행동을 먼저 둔다

토스 글은 `정보 구조`, `자동화`, `프로토타입` 같은 말을 쓰더라도 먼저 사람이 실제로 무엇을 하는지 보여준다.

과제 답변에서 바꿀 문장:

- `인지 부하를 줄였습니다` → `참석자가 매번 모든 일정을 떠올리지 않아도 되게 했습니다`
- `효율적인 조율 경험을 설계했습니다` → `주최자가 가능한 시간을 다시 묻고 표로 비교하는 일을 줄이고 싶었습니다`
- `AI 기반 추천을 제공했습니다` → `AI가 시간을 정해주는 대신, 왜 이 시간이 후보가 되는지 근거를 보여주게 했습니다`

## 5. 스토리텔링 장치

### 반전: 개인 문제에서 모두의 문제로

`매일 하던 업무를 디자인하기`는 개인 업무 도구가 팀의 문제로 확장되는 흐름을 가진다. 과제 답변에서도 `내가 본 불편`을 `여러 사용자 역할이 공유하는 구조적 문제`로 확장하면 좋다.

과제 답변 적용:

- 주최자의 번거로움은 참석자의 입력 부담과 연결된다.
- 참석자의 모호한 응답은 주최자의 재확인 노동으로 돌아온다.
- 선택 참석자를 배려하지 않는 추천은 회의 확정 이후의 불만이나 재조율로 이어진다.

### 순서 뒤집기: 산출물이 아니라 검증 환경

`AI로 바꾼 제품 설계의 순서`는 AI가 생산성을 높인 게 아니라 설계 순서를 바꿨다고 말한다. 답변에서도 AI를 기능으로 과시하지 말고 검증과 판단의 도구로 설명한다.

과제 답변 적용:

- `AI로 추천했습니다`보다 `AI를 이용해 충돌 케이스를 빠르게 드러내고, 사람이 결정해야 할 기준을 선명하게 만들었습니다`.
- `자동화했습니다`보다 `반복 비교를 줄여 주최자가 관계와 우선순위를 판단하는 데 집중하게 했습니다`.

### 물성: 추상 개념을 손에 잡히게 만든다

`디자이너가 시안 대신 앱을 만든 이유`는 움직임, 질감, 실제 기기에서 확인하는 과정을 통해 추상적 인터랙션을 구체화한다.

과제 답변 적용:

- 프로토타입을 만들었다면 `화면을 그렸다`가 아니라 어떤 순간을 실제로 확인했는지 쓴다.
- 예: `필수 참석자가 모두 불가한 상황`, `선호는 겹치지만 불가가 충돌하는 상황`, `미응답자가 있는 상황`을 실제 흐름에서 확인했다.

## 6. 토스식으로 보이는 문체 요소

### 자주 보이는 연결

- `처음에는 A라고 생각했어요. 그런데 B가 보였어요.`
- `보통은 A로 가잖아요. 저는 B를 선택했어요.`
- `문제는 A가 아니라 B였어요.`
- `그래서 질문을 바꿨어요.`
- `결국 중요한 건 A가 아니라 B였어요.`
- `이 방식은 이번 프로젝트뿐 아니라 다른 문제에도 적용할 수 있어요.`

과제 제출에서는 반말 느낌을 줄이고 다음처럼 바꾼다.

- `처음에는 A라고 봤습니다. 그런데 실제 흐름을 나눠보니 B가 더 큰 문제였습니다.`
- `보통은 A를 먼저 떠올리지만, 저는 B를 먼저 정리했습니다.`
- `그래서 질문을 바꿨습니다.`
- `결국 핵심은 시간을 추천하는 것이 아니라, 추천을 납득할 수 있게 만드는 것이었습니다.`

### 피해야 할 AI식 표현

- `사용자 니즈를 반영하여 직관적인 경험을 제공`
- `효율성과 편의성을 극대화`
- `최적의 솔루션을 도출`
- `사용자 중심 관점에서 문제를 해결`
- `원활한 커뮤니케이션을 지원`

이 표현들은 판단을 숨긴다. 반드시 구체적 행동으로 바꾼다.

## 7. 과제 답변용 변환 템플릿

### 문제 발견형

`처음에는 [표면 문제]가 핵심이라고 생각했습니다. 하지만 [사용자 역할]의 흐름을 따라가 보니, 더 큰 문제는 [반복되는 마찰]이었습니다. [과제 조건/관찰 근거] 때문에 이 마찰은 단순한 불편이 아니라 [결정/신뢰/시간/관계 비용]으로 이어졌습니다. 그래서 저는 문제를 [새로운 문제 정의]로 다시 정의했습니다.`

### 해결 설명형

`이 기준에서 [기능명]을 설계했습니다. 목적은 [기능이 하는 일]이 아니라 [사용자 행동 변화]였습니다. 예를 들어 [구체 상황]에서는 [제품 반응]을 보여주어, 사용자가 [판단/선택/이해]할 수 있게 했습니다. 반대로 [버린 대안]은 [버린 이유] 때문에 우선순위에서 낮췄습니다.`

### AI 활용형

`AI는 결정을 대신하는 역할로 두지 않았습니다. 대신 [반복 비교/충돌 탐지/후보 생성/문구 정리]처럼 사람이 놓치기 쉬운 부분을 빠르게 드러내는 장치로 사용했습니다. 최종 결정은 [사람이 판단해야 하는 기준]을 기준으로 남겨두었습니다.`

### 배운 점형

`이 과정을 통해 [주제]를 [기존 관점]이 아니라 [새 관점]으로 보게 됐습니다. 좋은 설계는 [기능적 결과]만 만드는 것이 아니라, 사용자가 [납득/안심/판단/조율]할 수 있는 기준을 함께 제공해야 한다고 봤습니다.`

## 8. 품질 체크리스트

- 첫 문단에 실제 장면이나 과제 조건이 있는가?
- `문제는 무엇인가`가 기능 없이 설명되는가?
- 질문을 바꾼 순간이 있는가?
- 선택한 해결책과 버린 대안이 함께 보이는가?
- AI를 썼다면 도구 과시가 아니라 판단과 검증의 변화로 설명했는가?
- `사용자 중심`, `직관적`, `효율적` 같은 추상어를 행동 언어로 바꿨는가?
- 마지막 문장이 `그래서 무엇을 배웠고, 무엇을 다르게 판단하게 됐는가`로 끝나는가?
