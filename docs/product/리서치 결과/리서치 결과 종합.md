# 리서치 결과 종합 — Claude × Codex 통합

> 이 문서는 `TOSS 프디 챌린지 2026` 리서치 폴더의 최종 확정본(버전 표시 제거)을 스타터킷으로 옮긴 스냅샷이다.
> 버전 히스토리와 원본(클로드/코덱스 개별 분석본 포함)은 원본 리서치 폴더에 그대로 남아있다.  
> 목적: 두 분석의 공통 결론과 차이를 확인하고, 원본 응답을 기준으로 문제 정의·가설·솔루션 방향을 한 번 더 통합한다.

## 1. 통합 결론

두 분석은 핵심적으로 같은 결론에 도달했다.

> 이번 조사에서 가장 두꺼운 근거는 `후보를 고르는 판단 부담`이 아니라, 캘린더의 불완전한 정보를 실제 참석 가능 상태로 바꾸기 위한 `취합·확인·리마인드 비용`이다.

따라서 현재 문제 정의를 다음의 이중 구조로 업데이트하는 방향이 가장 근거에 부합한다.

1. **핵심 문제 — 주최자의 확인·취합 비용:** 캘린더만으로 일정의 고정성·조정 가능성·숨은 불가를 알 수 없어 개별 확인과 후보 재생성이 반복된다.
2. **보조 문제 — 참석자의 관계적 비용:** 참석자는 비선호를 표현하면 전체 재조율을 유발할까 봐 침묵하고, 집중 단절과 업무 재조정 비용을 개인적으로 감수한다.

기존 UX Flow의 `일반 후보/조정 후보`, `미응답`, `불가/비선호`, `추천 이유` 구조는 폐기할 필요가 없다. 다만 **추천 알고리즘이 주인공이 아니라, 불확실한 조건을 최소 입력으로 확실하게 만드는 과정이 주인공**이 되어야 한다.

## 2. 두 분석의 공통점

| 공통 판단 | 근거 |
|---|---|
| 핵심 사용자는 주최자 | 수집·확인·재촉·확정 책임을 부담 |
| 가장 반복되는 비용은 취합·확인 | 공통 빈 시간 찾기 4/6, 추가 리마인드 5/6 |
| 캘린더의 busy/free는 최종 진실이 아님 | 일정의 조정 가능 여부와 미등록 부재를 알 수 없음 |
| 기존 핵심 문제인 ‘양보 판단 기준 부족’은 수정 필요 | 최종 후보 결정 0/6, 우선순위 판단 어려움 0/6 |
| 불가와 비선호는 구분할 가치가 있음 | 비선호 경험 6/7, 재조율 부담 때문에 요청하지 않음 4/7 |
| 참석 역할·이유 정보가 중요 | 판단에 필요한 정보 1위가 `내 참석 이유·역할` 4/7 |
| 미응답 리마인드는 직접적인 해결 기회 | 추가 메시지·리마인드 5/6 |
| 소표본과 후속 질문의 한계를 밝혀야 함 | n=7 편의표본, 비표준 후속 인터뷰 |

## 3. 두 분석의 차이

| 구분 | Claude 분석 | Codex 분석 | 통합 판단 |
|---|---|---|---|
| 분석 범위 | 응답자별 발화, 반례, H1~H5 대조가 상세함 | 문제 정의와 솔루션 인과를 더 압축적으로 판정함 | Claude의 근거 상세도 + Codex의 의사결정 구조를 결합 |
| 문제 정의 선택지 | 기존 문제 유지·흡수·이중 구조 세 가지 제안 | 핵심 문제를 확인·취합 비용으로 이동하도록 명확히 권고 | `핵심 A + 보조 B` 이중 구조 채택이 가장 안전 |
| 기존 Flow 평가 | 현재 정책 상당수가 사용자 요구와 겹친다고 봄 | 정책 방향은 유지하되 추천 중심 서사는 약화해야 한다고 봄 | Flow를 폐기하지 않고 제품 개입의 중심을 재배치 |
| 반례 | 시차로 선택지가 하나인 조직, 고정 회의 중심 조직을 상세 분석 | 표본 한계와 반증 조건에 초점 | 타깃 범위와 가설 반증 조건에 모두 반영 |
| 무응답 | 안 읽음/읽었지만 미정/보류 세분화 가능성 제안 | 필수 미응답 보류와 자동 리마인드에 집중 | 실제 연동 없는 MVP에서는 단순 상태로 구현, 세분화는 후속 범위 |
| 회의실 | 현실 제약으로 기록 | MVP 범위 밖으로 명확히 제외 | 제외하되 실제 사례에 근거한 범위 선택으로 설명 |

## 5. 통합 사용자 조사 결과

### 5.1 현재 행동

```text
캘린더에서 참석자들의 일정을 확인
→ 공통으로 비어 보이는 후보를 찾음
→ 일정이 찬 사람에게 이동 가능한 일정인지 확인
→ 캘린더에 없는 부재·비선호가 있는지 확인
→ 응답이 늦은 사람을 재촉
→ 새 정보가 생기면 후보를 다시 만듦
→ 최종 초대 발송
```

### 5.2 핵심 마찰

| 마찰 | 관찰 사실 | 손실 |
|---|---|---|
| 일정의 의미가 불명확함 | 고정 회의인지 조정 가능한 업무인지 구분 어려움 3/6 | 개별 확인 필요 |
| 필수 참석자 공통 시간이 부족함 | 2/6 | 범위 확대·기존 일정 조정 필요 |
| 응답 상태가 불확실함 | 리마인드 필요 5/6, 수락 미응답 발화 | 대기·감정 소모 |
| 비선호를 말하기 어려움 | 재조율 부담 4/7 | 개인 업무 재조정·집중 단절 |
| 참석 필요성이 불명확함 | 역할·이유를 필요로 함 4/7, 실제 제공 2/7 | 양보 여부 판단 어려움 |

### 5.3 결과 손실

- 의사결정 시기 지연 3/6
- 회의 확정 전 관련 업무 대기 2/6
- 디자인·개발 등 후속 업무 착수 지연 1/6
- 다른 일정이 먼저 잡혀 재조율 1/6
- 비선호 시간 참석 후 계획 업무 재조정 3/7
- 집중 재진입 비용 2/7

### 5.4 반례와 적용 범위

- 시차 때문에 선택지가 사실상 하나인 분산 조직에는 후보 비교·추천 가치가 작다.
- 고정 회의가 대부분인 조직에는 임시 회의 조율 빈도가 낮다.
- 빠른 응답을 강하게 강제하는 조직은 문제를 제품이 아니라 문화로 억누르고 있다.
- 회의실 부족이 더 큰 병목인 조직에서는 시간 추천만으로 문제를 해결할 수 없다.

따라서 1차 타깃은 **같은 시간대에서 일하고, 임시 협업 회의가 있으며, 여러 후보 사이에 일정 조정 여지가 있는 팀**으로 좁힌다.

## 6. 통합 문제 정의

### 핵심 사용자

4명 이상의 임시 업무 회의를 잡고 최종 시간을 확정해야 하는 주최자.

### 상위 목적

필요한 사람이 참여할 수 있는 시간을 적은 조율 비용으로 확정해 의사결정과 후속 업무를 진행한다.

### 핵심 문제

> 주최자는 캘린더에서 공통 빈 시간을 찾더라도 등록된 일정이 고정인지 조정 가능한지, 비어 있는 시간이 실제 가능한지 알 수 없다. 이 불확실성을 해소하기 위해 참석자에게 개별적으로 다시 묻고 응답을 재촉하며 후보를 반복 생성해야 하므로 일정 확정과 후속 의사결정이 늦어진다.

### 보조 문제

> 참석자는 가능한 시간이더라도 집중 업무를 보호하고 싶지만, 자신의 선호를 표현하면 여러 사람을 다시 조율하게 할까 봐 변경을 요청하지 않는다. 그 결과 일정은 확정되더라도 집중 단절과 업무 재조정 비용이 참석자 개인에게 남는다.

### 문제 우선순위 근거

| 기준 | 핵심 문제 A | 보조 문제 B |
|---|---|---|
| 반복성 | 공통 후보 탐색 4/6, 리마인드 5/6 | 비선호 경험 6/7 |
| 업무 영향 | 의사결정·후속 업무 지연 | 업무 재조정·집중 단절 |
| 현재 우회 비용 | Slack 개별 확인·재촉·후보 재생성 | 불편을 감수하고 그대로 참석 |
| 제품 개입 가능성 | 불확실한 조건만 수집·상태화 가능 | 비동기 비선호 입력·추천 반영 가능 |
| MVP 우선순위 | 핵심 | 보조 |

## 7. 통합 제품 가설

### 핵심 가설

> 캘린더에서 성립 가능한 후보를 먼저 좁히고 참석자에게 전체 일정을 다시 입력시키는 대신 `숨은 불가·비선호`와 `겹친 일정의 조정 가능 여부`만 확인하게 하면, 주최자의 개별 확인·리마인드·후보 재생성 횟수가 줄어 회의를 더 빠르게 확정할 수 있을 것이다.

### 보조 가설

> 참석 이유와 후보별 남은 조정 비용을 함께 제공하면 참석자는 자신의 일정 양보 여부를 판단하고, 주최자는 필수 참여를 보호하면서 비선호 희생이 적은 시간을 선택할 수 있을 것이다.

### 제품과 사람의 권한

- 제품: 후보 탐색, 불확실성 수집, 미응답 안내, 추천 이유·남은 양보 설명
- 참석자: 숨은 불가·비선호·일정 조정 가능성 판단
- 주최자: 회의 목적에 필요한 역할 지정, 최종 시간 확정
- 제품이 하지 않는 것: 기존 일정이 이동 가능하다고 임의로 단정하거나 사람 대신 자동 확정

### 반증 조건

1. 제품 후보를 본 뒤에도 사용자가 메신저에서 전원에게 다시 확인한다.
2. 참석자 응답 완료율이 기존 Slack 방식보다 낮다.
3. 조정 가능 여부 입력이 이해되지 않거나 실제 일정 변경 완료로 오해된다.
4. 주최자가 필수·권장·선택을 회의 목적 기준으로 구분하지 못한다.
5. 추천 이유를 본 뒤에도 후보 차이와 선택 대가를 설명하지 못한다.

## 8. 통합 솔루션 방향과 우선순위

| 우선순위 | 방향 | 직접 해결하는 문제 | MVP |
|---:|---|---|---|
| 1 | 캘린더 기반 후보를 먼저 필터링 | 반복적인 시간 탐색 | 포함 |
| 2 | 숨은 불가·비선호·일정 조정 가능성만 추가 수집 | 캘린더 불확실성 | 포함 |
| 3 | 응답 마감·자동 리마인드·필수 미응답 보류 | 수동 재촉·거짓 합의 | 포함 |
| 4 | 후보별 확실/확인 필요/비선호/미응답 표시 | 후보 유효성 판단 | 포함 |
| 5 | 참석 역할과 `왜 필요한지` 제시 | 참석자의 양보 판단 | 포함 |
| 6 | 성립 가능한 후보 1개 추천 + 대안 1~2개와 대가 | 선택 부담 | 축소 포함 |
| 7 | 안 읽음/읽음/보류 세분화 | 무응답 상태 해석 | 후속 |
| 8 | 회의실·부분 참석·시차 최적화 | 다른 병목 | 제외 |

핵심 흐름은 다음으로 압축한다.

```text
회의 목적·참석 역할 설정
→ 캘린더 기반 후보 필터링
→ 불확실한 조건만 참석자에게 확인
→ 미응답 리마인드
→ 성립 후보와 남은 양보 설명
→ 주최자가 최종 확정
```

## 9. 검증 계획

| 검증 질문 | 행동 지표 | 실패 조건 |
|---|---|---|
| 참석자는 무엇을 입력해야 하는지 이해하는가? | 도움 없이 불가·비선호·조정 가능 응답 완료 | 기존 캘린더 일정을 다시 입력하려 함 |
| 주최자는 후보의 불확실성을 이해하는가? | 필수 미확정 후보를 확정하지 않음 | 미응답·확인 중을 가능으로 오해 |
| 추천은 실제 판단을 돕는가? | 추천 이유와 후보의 대가를 설명 | 추천을 무시하고 캘린더를 처음부터 재비교 |
| 조정 가능과 변경 완료를 구분하는가? | 확정 후 필요한 다음 행동을 말함 | `조정 가능`을 이미 변경된 상태로 오해 |
| 관계적 부담을 줄이는가? | 비선호를 직접 연락 없이 입력 | 비선호 입력을 꺼리거나 거짓 응답 |

측정 후보는 확정까지 걸린 시간, 주최자의 추가 확인 횟수, 리마인드 횟수, 후보 재생성 횟수, 과업 성공률, 판단 확신도다. 현재 조사에는 기준값이 없으므로 개선 수치를 임의로 약속하지 않는다.

## 10. 조사 한계

- n=7 지인 편의표본이며 전체 직장인을 대표하지 않는다.
- 주최만 담당하는 응답자는 없고 참석 위주 또는 혼합 역할이다.
- 후속 인터뷰는 공통 질문지보다 응답별 즉흥 질문이 많다.
- 일부 질문은 원하는 솔루션을 직접 물어 응답을 유도했을 수 있다.
- 설문 선택지가 문제 후보를 미리 구조화했기 때문에 새로운 문제를 놓쳤을 수 있다.
- 실제 행동 관찰이 아니라 최근 경험 회상과 자기보고가 중심이다.
