# 프로젝트 정의 — 모두의 한 시간 MVP

> 이 문서는 `TOSS 프디 챌린지 2026` 리서치 폴더의 최종 확정본(버전 표시 제거)을 스타터킷으로 옮긴 스냅샷이다.
> 버전 히스토리와 원본은 원본 리서치 폴더에 그대로 남아있다.  
> 기준 답변: [과제 질문 3가지](../과제%20질문%203가지/과제%20질문%203가지.md)  
> 기준 정책: [제품 데이터 정책](../정책/제품%20데이터%20정책.md)  
> 기준 프로토타입: [참고 프로토타입](../../../src/externalComponents/참고%20프로토타입/README.md)

## 0. 이전 버전 대비 변경 요약

| 구분 | 이전 | 최신 |
|---|---|---|
| 문제 초점 | 조건 입력과 후보 결과의 단절 | 탐색, 실제 가능 여부 검증, 최소 조정안 판단 비용 |
| 추천 단위 | 후보 하나를 선택해 요청 정보 작성 | 후보 3개 묶음을 참석자에게 검토 요청 |
| 조정 전략 | 기간 확장, 길이 단축, 참석 구분 재검토까지 넓게 제안 | 현재 조건 안의 조정안과 조건 변경 제안을 분리 |
| AI 역할 | 조건 조정과 조정 전략 제안 | 후보 차이, 대가, 조정 요청 가능 근거를 설명하되 자동 확정 금지 |
| 기준 정책 | 제품 데이터 정책 초안 | 제품 데이터 정책 정리본 |

## 1. 과제 조건과 제품 가정

### 공식 과제 조건

- 같은 회사 동료 6명이 다음 주 안에 1시간 회의를 잡는다.
- 사람마다 일정과 선호 조건이 다르다.
- 필수 참석자와 선택 참석자가 존재한다.
- 사용자가 동료들의 상황을 파악하고 적절한 시간을 확정한다.

### 제품 환경 가정

- 주최자 1명은 기본 포함되어 있으며, 주소록에서 추가 참석자 5명을 선택한다.
- 회사 캘린더의 일정 시간과 일정 제목은 후보 탐색의 출발점으로 사용한다.
- 캘린더의 빈 시간은 실제 가능을 보장하지 않고, 일정 있음은 절대 불가를 보장하지 않는다.
- 제출용 MVP에서는 실제 캘린더 연동 대신 샘플 데이터와 프로토타입으로 시뮬레이션한다.

## 2. 핵심 사용자와 목적

| 사용자 | 이루려는 것 | 제품이 줄여야 하는 비용 |
|---|---|---|
| 주최자 | 필요한 사람들이 참석할 수 있는 시간을 찾아 회의를 확정 | 공통 빈 시간 탐색, 개별 확인, 조정안 비교 |
| 필수 참석자 | 회의 목적과 자신의 역할을 알고 참석·조정 가능 여부를 판단 | 불필요한 전체 일정 입력, 모호한 양보 요청 |
| 선택 참석자 | 참여 필요도와 후보별 영향을 이해하고 가능한 범위에서 응답 | 조용히 희생되거나 배제되는 비용 |
| 그룹 | 필요한 사람이 필요한 논의 시점에 참여해 회의가 실제로 진행 | 거짓 합의, 미응답 상태의 확정, 반복 재조율 |

## 3. 문제 정의

### 핵심 문제

> 회의 조율에서 주최자는 캘린더와 메신저를 오가며 공통 빈 시간을 찾고, 그 시간이 실제 가능한지 확인하고, 후보가 맞지 않을 때 어떤 조정안이 가장 적은 비용인지 판단해야 한다. 이 반복 비교와 확인 비용은 회의 확정을 늦추고, 참석자에게는 모호한 양보나 침묵 비용을 만든다.

### 문제를 세 부분으로 나눈 이유

| 문제 | 왜 먼저 풀어야 하는가 |
|---|---|
| 공통 빈 시간 탐색 비용 | 공통 빈 시간을 찾지 못하면 검토 요청과 조정도 시작할 수 없다 |
| 실제 가능 여부 검증 비용 | 캘린더만으로는 일정 고정 여부, 비선호, 숨은 제약을 알 수 없다 |
| 최소 조정안 의사결정 비용 | 후보가 없거나 충돌할 때 누가 어떤 비용을 감수해야 하는지 비교해야 한다 |

## 4. 제공할 가치

> 주최자가 반복 비교와 개별 확인에 쓰던 시간을 줄이고, 참석자 응답과 조정 비용을 반영해 납득 가능한 한 시간을 확정하게 한다.

- **주최자:** 참석자 기준, 일정 범위, 미팅 길이, 후보 결과를 같은 화면에서 조정하고 후보 묶음을 검토 요청할 수 있다.
- **참석자:** 전체 일정을 다시 입력하지 않고 후보별 가능, 비선호, 조정 필요, 불가만 응답한다.
- **그룹:** 필수 참석자 성립 가능성을 우선 보면서도 선택 참석자 불참, 비선호, 조정 요청 비용을 숨기지 않는다.

## 5. 제품 가설

### 핵심 가설

> 참석자 기준, 시간 조건, 후보 결과를 한 화면에서 탐색하고, 후보를 바로 확정하지 않고 참석자 검토 요청을 거치면 주최자는 더 적은 확인 비용으로 실제 성립 가능한 회의 시간을 찾을 수 있을 것이다.

### 보조 가설

- 후보 3개를 묶어 검토 요청하면 하나의 시간을 확정하기 전 숨은 조건을 비교할 수 있다.
- `가능`, `비선호지만 가능`, `조정 필요`, `불가`를 분리하면 단순 가능/불가보다 주최자의 선택 대가가 선명해진다.
- 필수 참석자의 겹친 일정이 다른 회의에서 선택 참석인 경우만 조정안으로 다루면, 제품이 알 수 없는 조정 가능성을 과도하게 추론하지 않는다.

## 6. 설계 원칙

1. **탐색은 빠르게, 확정은 검증 뒤에 한다.**
2. **추천은 확정이 아니라 검토 후보 묶음이다.**
3. **필수 참석자의 공통 가능 여부를 후보 성립의 1순위 기준으로 둔다.**
4. **선택 참석자 충돌은 숨기지 않고 후보의 대가로 설명한다.**
5. **조정안과 조건 변경 제안은 분리한다.**
6. **참석자의 미응답, 확인 중, 조정 필요는 가능으로 간주하지 않는다.**
7. **AI는 반복 비교와 설명을 돕고, 최종 확정은 주최자가 한다.**

상세 데이터 상태, 추천, 조정안, 캘린더 표시, AI 사용 기준은 [제품 데이터 정책](../정책/제품%20데이터%20정책.md)을 따른다.

## 7. MVP 범위

### 포함

- 주최자 포함 6명 회의 시나리오
- 참석자 요약과 참석자 설정 모달
- 일정 범위, 미팅 길이, 탐색 시간 설정
- 가능한 일정 리스트와 캘린더뷰
- 모두 참석 후보 2개와 빠른 대안 1개를 포함한 후보 묶음
- 요청 정보 작성: 회의 목적, 응답 기한, 첨부 자료, 참석자별 필요한 이유
- 참석자 응답 상태: 가능, 비선호지만 가능, 조정 필요, 불가
- 미응답, 조정 필요, 비선호를 분리한 응답 현황
- 필수 참석자 공통 시간이 없을 때 조정안과 조건 변경 제안 분리
- AI 조정안 검토와 주최자 최종 확정

### 제외

- 실제 캘린더·메신저 API 연동
- 실제 알림·읽음 상태 추적
- 실제 다중 사용자 백엔드
- 모바일 최적화
- 회의실, 시차, 반복 일정
- 확정 후 전체 재조율 자동화
- 상용 개인정보·권한 정책

## 8. 검증·반증 조건

| 검증 질문 | 성공 신호 | 실패 조건 |
|---|---|---|
| 주최자는 한 화면에서 조건과 결과의 관계를 이해하는가? | 조건을 바꾸면 후보가 달라진다고 설명함 | 결과와 조건을 별개 영역으로 이해함 |
| 후보가 확정이 아니라 검토 요청 대상임을 이해하는가? | 후보 선택 후 요청 정보 작성이 필요하다고 말함 | 추천 후보를 바로 확정된 일정으로 오해함 |
| 응답 상태의 차이를 이해하는가? | 비선호, 조정 필요, 불가를 다른 대가로 설명함 | 조정 필요를 가능으로 처리함 |
| 필수 참석자 공통 없음 상태에서 조정안과 조건 변경을 구분하는가? | 현재 조건 내 조정 요청과 조건 자체 변경을 구분함 | 기간 확장·길이 단축을 조정안으로 오해함 |
| AI 조정안의 권한 경계를 이해하는가? | AI는 제안하고 주최자가 결정한다고 설명함 | AI가 자동 확정한다고 느낌 |

## 9. 근거와 한계

- 설문 7명, 후속 인터뷰 5명의 지인 편의표본이다.
- 공통 빈 시간 탐색과 추가 확인 비용은 리서치에서 반복 관찰됐지만, 통합 탐색 구조가 실제로 시간을 줄이는지는 프로토타입 검증이 필요하다.
- 후보 3개 묶음과 빠른 대안은 설계 가설이며, 후보 수와 노출 방식은 테스트로 조정해야 한다.
- 조정안 범위는 과도한 추론을 막기 위해 좁혔고, 실제 서비스에서는 조직의 권한·캘린더 데이터·개인정보 정책에 따라 달라질 수 있다.
