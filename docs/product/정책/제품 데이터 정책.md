> 이 문서는 `TOSS 프디 챌린지 2026` 리서치 폴더의 최종 확정본(버전 표시 제거)을 스타터킷으로 옮긴 스냅샷이다.
> 버전 히스토리와 원본은 원본 리서치 폴더에 그대로 남아있다.  
> 문서 상태: 참고 프로토타입 이후 추천/조정 로직 정리본  
> 기준 문서: `제품 데이터 정책 초안`, [프로젝트 정의](../프로젝트%20정의/프로젝트%20정의.md), [UX flow](../UX%20flow/UX%20flow.md), [참고 프로토타입](../../../src/externalComponents/참고%20프로토타입/README.md)

## 1. 변경 요약

이 정리본에서는 통합 탐색 화면의 추천 결과와 조정안 정책을 새로 정의한다. 핵심 변경은 다음과 같다.

| 구분 | 초안 | 정리본 |
|---|---|---|
| 추천 단위 | 후보 하나를 선택하는 흐름 중심 | 후보 3개를 묶어 참석자에게 검토 요청 |
| 빠른 대안 | 추천 후보와 구분이 약함 | 모두 참석 후보 2개 + 가장 빠른 대안 1개로 명시 |
| 후보 없음 | 기간 확장, 길이 단축, 참석 구분 변경을 조정안처럼 제안 | 현재 조건 내 조정안과 조건 변경 제안을 분리 |
| 조정안 | AI가 안건 순서, 부분 참석, 회의록 공유 등을 제안 가능 | 이번 범위에서는 `필수 참석자의 겹친 일정이 다른 회의에서 선택 참석인 경우`만 조정안 |
| 캘린더 표시 | 후보 상태 중심 | 빗금, 옅은 표시, 진한 표시, 상세 보기 정책을 분리 |
| AI 범위 | 조정 전략 제안 범위가 넓음 | 판단할 수 없는 참여 필요도, 안건 순서, 이동 가능성은 추론 금지 |

## 2. 문서 목적

이 문서는 회의 조율 MVP에서 어떤 데이터를 입력받고, 어떤 상태로 해석하며, 어떤 기준으로 후보와 조정안을 보여주고, 어떤 정보는 숨길지 정리한다.

화면 UI 문구보다 한 단계 아래, 실제 구현 데이터 모델보다 한 단계 위의 정책 문서다. 프로토타입에서는 이 문서를 기준으로 샘플 데이터, 상태값, 후보 추천 문구, 캘린더 표시, 예외 화면을 만든다.

## 3. 정책 원칙

1. **추천은 확정이 아니라 검토 후보 묶음이다.**  
   주최자는 후보 3개를 골라 참석자에게 확인을 요청한다. 제품은 후보의 장단점을 설명하지만, 최종 확정 권한은 주최자에게 둔다.

2. **필수 참석자의 공통 가능 여부가 후보 성립 기준이다.**  
   필수 참석자 그룹이 함께 참석할 수 없는 시간은 일반 후보가 될 수 없다. 선택 참석자의 충돌은 후보 제외가 아니라 대안 비용으로 설명한다.

3. **빠른 대안은 선택 참석자 비용을 드러내는 비교 후보다.**  
   모두 참석 가능한 후보가 있어도, 일정상 더 빠른 시간이 필수 참석자는 가능하고 선택 참석자 일부만 불가능하다면 `빠른 대안`으로 함께 보여줄 수 있다.

4. **조정안과 조건 변경은 다른 개념이다.**  
   조정안은 현재 조건 안에서 참석 가능성을 만들 수 있는 근거가 있는 경우다. 일정 범위 확장, 미팅 길이 단축, 필수/선택 재설정은 조건 변경이다.

5. **조정 가능성은 추론하지 않는다.**  
   캘린더 일정 제목, 참석자 역할, 회의 맥락만 보고 일정 이동 가능성이나 부분 참석 가능성을 단정하지 않는다.

6. **상세 일정 보기는 왜 안 되는지 설명해야 한다.**  
   축약 캘린더에서 빗금으로 표시된 시간은 상세 보기에서 해당 참석자의 실제 겹친 일정을 보여준다. 빈칸이나 `일정 없음`으로 처리하지 않는다.

7. **참석자의 응답은 동의가 아니라 판단 정보다.**  
   `조정 요청 가능`은 조정 요청을 보낼 수 있다는 뜻이지, 참석자가 이미 조정에 동의했다는 뜻이 아니다.

## 4. 주요 데이터 객체

| 객체 | 의미 | 생성/수정 주체 | 보이는 주요 화면 |
|---|---|---|---|
| `MeetingDraft` | 아직 확정되지 않은 회의 초안 | 주최자 | 미팅 일정 탐색, 미팅 정보 작성 |
| `Participant` | 회의 대상자. 주최자 포함 6명 | 주최자 | 미팅 조건, 참석자 선택 모달, 응답 현황 |
| `ParticipantRole` | `필수` 또는 `선택` 참석 구분 | 주최자 | 미팅 조건, 후보 카드, 응답 요청 |
| `TimeCondition` | 미팅 길이, 일정 범위, 탐색 시간 | 주최자 | 미팅 조건 |
| `CalendarEvent` | 연동 캘린더에서 가져온 일정 구간, 제목, 참석자, 참석 역할 | 시스템 | 캘린더, 상세 일정 보기 |
| `CandidateSlot` | 조건에 따라 계산된 특정 시간 후보 | 시스템 | 추천 일정, 캘린더 |
| `CandidateSet` | 참석자에게 함께 검토 요청할 후보 묶음 | 시스템/주최자 | 추천 일정, 검토 요청 발송 |
| `CandidateReason` | 후보가 추천/대안/제외/조정안이 된 이유 | 시스템 | 후보 카드, 상세 보기 |
| `AdjustmentProposal` | 현재 조건 안에서 조정 요청을 보낼 수 있는 근거가 있는 안 | 시스템/AI 보조, 주최자 선택 | 필수 공통 없음, 조정 검토 |
| `ConditionChangeSuggestion` | 현재 조건을 바꿔 다시 탐색하는 제안 | 시스템/AI 보조 | 필수 공통 없음, 조건 조정 |
| `InvitationContext` | 회의명, 목적, 자료, 참석자별 필요한 이유 | 주최자/AI 보조 | 미팅 정보 작성, 참석자 응답 |
| `Response` | 참석자의 후보별 응답 상태 | 참석자 | 응답 현황, 최종 확정 |

## 5. 상태 정의

### 5.1 회의 초안 상태

| 상태 | 의미 | 다음 행동 |
|---|---|---|
| `exploring` | 참석자와 시간 조건을 입력하고 가능한 일정을 탐색 중 | 조건 수정, 후보 보기 |
| `candidate_set_ready` | 참석자에게 보낼 후보 묶음이 준비됨 | 후보 확인, 미팅 정보 작성 |
| `request_ready` | 검토 요청에 필요한 미팅 정보가 준비됨 | 검토 요청 발송 |
| `collecting_responses` | 후보 3개에 대한 참석자 응답을 기다리는 중 | 응답 확인, 리마인드 |
| `needs_adjustment` | 후보 응답 결과 조정 또는 조건 변경이 필요함 | 조정안 검토, 조건 변경 |
| `adjustment_request_ready` | 조정 요청을 보낼 대상과 근거가 정리됨 | 조정안 검토 요청 발송 |
| `collecting_adjustment_responses` | 조정 요청 응답을 기다리는 중 | 응답 확인 |
| `confirmed` | 주최자가 최종 시간을 확정함 | 확정 안내 |

### 5.2 후보 시간 상태

| 상태 | 의미 | 후보 카드 처리 |
|---|---|---|
| `all_available_recommended` | 필수/선택 참석자가 모두 가능한 후보 | 추천 후보로 우선 표시 |
| `all_available_option` | 모두 가능하지만 추천 1순위는 아닌 후보 | 비교 후보로 표시 |
| `fast_alternative` | 필수 참석자는 모두 가능하지만 선택 참석자 일부가 불가능한 이른 시간 | `빠른 대안`으로 표시 |
| `required_blocked` | 필수 참석자 그룹이 함께 참석할 수 없는 시간 | 일반 후보 제외, 캘린더 빗금 표시 |
| `optional_blocked` | 필수 참석자는 가능하지만 선택 참석자 일부 또는 전체가 불가능한 시간 | 옅은 표시 또는 빠른 대안 |
| `adjustment_candidate` | 필수 참석자 충돌이 있으나 조정 요청 가능 근거가 있는 시간 | 조정안 카드로 표시 |
| `no_candidate` | 현재 조건에서 보낼 확정 후보가 없음 | 조정안 또는 조건 변경 제안 |

### 5.3 참석자 응답 상태

| 상태 | 의미 | 후보 반영 |
|---|---|---|
| `available` | 후보 시간 참석 가능 | 후보 신뢰도 상승 |
| `available_but_not_preferred` | 참석은 가능하지만 피하고 싶은 시간 | 확정을 막지는 않지만 대가로 표시 |
| `needs_adjustment` | 가능성은 있으나 일정 조정 확인 필요 | 확정 전 추가 확인 대상 |
| `unavailable` | 참석 불가 | 필수 참석자면 후보 제외 또는 조정 필요 |
| `pending` | 아직 응답 없음 | 필수 참석자면 확정 신뢰도 낮춤 |

## 6. 시간 조건 정책

| 항목 | 명칭 | 입력 방식 | 기본값 |
|---|---|---|---|
| 미팅 길이 | `미팅 길이` | 프리셋 + 직접 입력 | 1시간 |
| 일정 범위 | `일정 범위` | 프리셋 + 직접 입력 | 이번 주 또는 최근 사용값 |
| 탐색 시간 | `탐색 시간` | 시작/종료 시간 입력, 30분 단위 제안 | 오전 9:00 ~ 오후 5:00 |

정책:

- `조회 기간`, `조회 시간`보다 `일정 범위`, `탐색 시간`을 우선한다. 주최자가 데이터를 검색한다기보다 회의 가능한 범위를 지정하는 행동에 가깝기 때문이다.
- 미팅 길이와 일정 범위는 프리셋을 먼저 제공하되, 직접 입력을 허용한다.
- 탐색 시간은 업무 시간 안에서 후보를 찾기 위한 조건이다. 특정 시간 제외 기능은 이번 범위에서 제외한다.

## 7. 추천 카드 구성 정책

### 7.1 기본 추천

추천 결과는 최대 3개 후보를 한 묶음으로 보여준다.

| 카드 유형 | 표시 조건 | 카드에 보여줄 정보 |
|---|---|---|
| `추천` | 모두 참석 가능한 후보 중 가장 적합한 시간 | 날짜/시간, 모두 참석 가능, 추천 이유 |
| `모두 가능` | 모두 참석 가능한 추가 후보 | 날짜/시간, 모두 참석 가능, 추천 후보와의 차이 |
| `빠른 대안` | 필수 참석자는 모두 가능하지만 선택 참석자 일부가 불가능한 이른 시간 | 날짜/시간, 필수 가능 수, 선택 불가 수, 대안 비용 |

추천 카드에는 다음 정보를 포함한다.

- 날짜와 시간
- 필수 참석자 가능 수
- 선택 참석자 가능 수
- 추천/모두 가능/빠른 대안 라벨
- 선택 시 발생하는 대가
- 정보 아이콘 또는 상세 보기 진입점

### 7.2 시나리오 2: 빠른 대안 포함

조건:

- 필수 참석자 3명 모두 가능한 시간이 존재한다.
- 선택 참석자까지 모두 가능한 후보가 2개 이상 있다.
- 그보다 빠르거나 일정상 의미 있는 시간 중 필수 참석자는 가능하지만 선택 참석자 1명이 불가능한 시간이 있다.

정책:

1. 모두 참석 가능한 후보 2개를 먼저 추천 묶음에 포함한다.
2. 가장 빠른 대안 1개를 함께 포함한다.
3. 빠른 대안은 `확정 추천`이 아니라 `선택 참석자 1명 불참 비용이 있는 대안`으로 표시한다.
4. 참석자에게는 3개 후보를 함께 검토 요청한다.
5. 주최자에게는 빠른 대안을 선택할 경우 누가 불참하는지, 필수 참석자는 모두 가능한지 명확히 보여준다.

### 7.3 시나리오 3: 필수 공통 없음

조건:

- 현재 미팅 길이, 일정 범위, 탐색 시간 안에서 필수 참석자 3명이 모두 함께 참석 가능한 시간이 없다.

정책:

1. 이 상태에서는 참석자에게 보낼 확정 후보가 없다고 판단한다.
2. 먼저 현재 조건 안에서 성립 가능한 `AdjustmentProposal`이 있는지 찾는다.
3. 조정안이 있으면 `가능하게 만드는 조정안`으로 보여준다.
4. 조정안이 없으면 `ConditionChangeSuggestion`으로 일정 범위, 미팅 길이, 참석자 역할 변경을 제안한다.
5. 조정안과 조건 변경 제안은 같은 카드 그룹 안에 섞지 않는다.

## 8. 조정안 정책

### 8.1 이번 범위에서 허용하는 조정안

이번 MVP 범위에서 조정안으로 인정하는 케이스는 하나다.

| 조정안 | 성립 조건 | 표시 문구 |
|---|---|---|
| 다른 회의의 선택 참석 일정 조정 요청 | 필수 참석자의 겹친 일정이 다른 회의에서 `선택 참석자`로 들어가 있는 경우 | `조정 요청 가능` |

예시:

- 이 회의에서는 김민준이 필수 참석자다.
- 후보 시간에 김민준의 다른 회의가 겹쳐 있다.
- 그 다른 회의에서 김민준은 선택 참석자다.
- 따라서 주최자는 김민준에게 해당 일정 조정 가능 여부를 확인 요청할 수 있다.

주의:

- `조정 요청 가능`은 참석자가 이미 조정에 동의했다는 뜻이 아니다.
- 제품은 조정 요청 근거를 보여주고, 주최자가 요청을 보낼지 결정한다.
- 참석자는 요청을 보고 가능/불가/조정 필요로 응답한다.

### 8.2 조정안으로 인정하지 않는 것

| 제외 항목 | 제외 이유 |
|---|---|
| 선택 참석자 1명만 불가하므로 회의록 공유로 대체 | 필수 참석자 공통 없음 문제의 조정안이 아니며, 선택 참석자 대체 전략에 가깝다 |
| 필수 참석자가 회의 앞부분만 필요하므로 안건 순서 조정 | AI가 이 시점에 참여 필요 구간을 판단할 수 없다 |
| 일정 이동 가능 표시가 있는 경우 | 이번 범위에서 이동 가능 데이터를 수집하지 않는다 |
| 내부 회의이고 본인이 주최자/필수가 아닌 경우 | 실제 조정 권한을 알 수 없고, 위 허용 케이스와 구분 실익이 낮다 |
| 미팅 길이 줄이기 | 현재 조건을 바꾸는 행동이므로 조건 변경 제안이다 |
| 일정 범위 늘리기 | 현재 조건을 바꾸는 행동이므로 조건 변경 제안이다 |
| 필수 참석자를 선택 참석자로 바꾸기 | 회의 성립 기준을 바꾸는 행동이므로 조건 변경 제안이다 |

## 9. 캘린더 표시 정책

### 9.1 축약 캘린더

| 표시 | 의미 | 상세 보기에서 보여줄 것 |
|---|---|---|
| 진한 색 | 추천 후보 또는 조정안 위치 | 후보 순위, 후보 유형, 관련 일정 |
| 빗금 | 필수 참석자 그룹이 모두 함께 참석할 수 없는 시간 | 어떤 필수 참석자의 어떤 일정 때문에 막혔는지 |
| 옅은 색 | 필수 참석자는 모두 가능하지만 선택 참석자 일부 또는 전체가 불가능한 시간 | 선택 참석자 충돌 정보 |
| 빈칸 | 현재 조건에서 뚜렷한 후보/충돌 표시가 필요 없는 시간 | 상세 보기에서도 실제 일정이 있으면 표시 가능 |

빗금의 의미는 `필수 참석자 3명이 모두 각각 불가능하다`가 아니라, `필수 참석자 3명이 함께 참석하는 후보로 성립하지 않는다`이다. 즉 필수 참석자 중 한 명이라도 겹친 일정이 있으면 빗금 표시 대상이 될 수 있다.

### 9.2 상세 일정 보기

정책:

- 상세 일정 보기를 켜면 참여 인원에 포함된 사람의 일정만 보여준다.
- 참여 인원이 아닌 사람의 일정은 샘플 데이터에 있어도 표시하지 않는다.
- 축약 캘린더에서 빗금으로 표시된 시간은 상세 보기에서 반드시 해당 충돌 일정을 보여준다.
- 일정이 있는 칸에 `일정 없음`을 표시하지 않는다.
- 일정 제목은 후보 판단에 필요한 범위로만 표시한다.
- 민감한 사유를 추론하거나 요약하지 않는다.

## 10. 정보 노출/숨김 정책

| 정보 | 주최자에게 노출 | 참석자에게 노출 | 정책 |
|---|---|---|---|
| 참석자 이름 | 노출 | 본인과 회의 맥락에 필요한 범위 | 회의 대상 확인에 필요 |
| 필수/선택 구분 | 노출 | 본인 역할로 노출 | 참석 이유와 함께 설명해야 함 |
| 후보 3개 | 노출 | 응답 요청 시 노출 | 후보별 응답을 받기 위해 필요 |
| 캘린더 일정 제목 | 후보 판단에 필요한 범위로 노출 | 본인 일정 중심 | 조정 가능성 추론에 쓰지 않음 |
| 선택 참석자 불참 비용 | 노출 | 후보별 영향으로 노출 | 선택 참석자 조건을 무시하지 않기 위해 필요 |
| 조정 요청 가능 근거 | 노출 | 요청 대상에게 노출 | 동의가 아니라 확인 요청임을 표시 |
| 비선호 응답 | 주최자에게 대가로 노출 | 본인 응답으로 노출 | 불가와 구분 |
| 참석자 메모 | 주최자에게 노출 | 본인 입력 확인 | 전체 참석자에게 공유하지 않음 |

## 11. AI 사용 정책

### 11.1 AI가 할 수 있는 것

- 후보 없음 원인을 요약한다.
- 후보 카드의 차이와 대가를 설명한다.
- 빠른 대안의 장단점을 정리한다.
- `필수 참석자의 겹친 일정이 다른 회의에서 선택 참석자인 경우`를 찾아 조정 요청 가능성을 제안한다.
- 일정 범위 확장, 미팅 길이 단축, 필수/선택 재설정 같은 조건 변경 제안을 만든다.
- 검토 요청 문구 초안을 작성한다.

### 11.2 AI가 하면 안 되는 것

- 참석자의 일정 이동 가능성을 단정한다.
- `조정 요청 가능`을 `조정 가능` 또는 `참석 가능`으로 바꾼다.
- 안건 순서나 부분 참석 가능성을 판단한다.
- 회의록 공유로 선택 참석자 불참을 대체해도 된다고 단정한다.
- 필수 참석자의 역할을 주최자 확인 없이 낮춘다.
- 주최자 확인 없이 후보를 자동 확정한다.
- 일정 제목만 보고 민감한 사유나 참석자의 의도를 해석한다.

## 12. 예외 상황 처리

| 예외 | 정책 | 주 사용자 행동 |
|---|---|---|
| 모두 참석 가능한 후보가 3개 이상 | 빠른 대안이 있으면 상위 2개 + 빠른 대안 1개, 없으면 모두 참석 후보 3개로 구성 | 후보 3개 검토 요청 |
| 모두 참석 가능한 후보가 1개뿐 | 1개 추천 후보 + 조건상 의미 있는 대안 후보를 함께 보여줌 | 후보 비교 후 요청 |
| 필수 참석자 공통 시간이 없음 | 조정안이 있는지 먼저 확인 | 조정안 검토 |
| 조정안도 없음 | 조건 변경 제안으로 전환 | 일정 범위/길이/역할 조정 |
| 선택 참석자만 불가 | 후보 제외하지 않고 대가로 표시 | 빠른 대안 검토 |
| 조건 변경 후 결과가 오래됨 | 기존 결과가 이전 조건 기준임을 표시 | 다시 탐색 |
| 참석자 미응답 | 확정 전 리마인드 또는 직접 확인 대상으로 분리 | 리마인드 |
| 응답 후 조정 필요 | 가능으로 처리하지 않음 | 추가 확인 또는 대안 선택 |

## 13. 화면별 데이터 맵

| 화면/장면 | 읽는 데이터 | 쓰는 데이터 | 파생 데이터 |
|---|---|---|---|
| 미팅 일정 탐색 초기 | `Participant`, `TimeCondition`, `CalendarEvent` | `ParticipantRole`, `TimeCondition` | 조건 요약 |
| 가능한 일정 보기 | `ParticipantRole`, `TimeCondition`, `CalendarEvent` | `CandidateSet` | 후보 상태, 추천 이유 |
| 상세 일정 보기 | `CandidateSlot`, `CalendarEvent`, `Participant` | 없음 | 충돌 사유, 일정 제목 |
| 필수 공통 없음 | `CandidateSlot` 계산 결과, `CalendarEvent` | 없음 | 조정안 또는 조건 변경 제안 |
| 조정안 검토 | `AdjustmentProposal`, `CalendarEvent` | 조정 요청 대상 선택 | 조정 요청 문구 |
| 미팅 정보 작성 | `CandidateSet`, 참석자 | `InvitationContext` | 참석자별 요청 문구 |
| 검토 요청 발송 | `CandidateSet`, `InvitationContext` | 요청 발송 상태 | 후보별 응답 링크 |
| 참석자 응답 | `InvitationContext`, 후보 3개, 본인 일정 | `Response` | 응답 완료 여부 |
| 응답 현황 | 전체 `Response` | 리마인드 요청 | 미응답/조정 필요/비선호 요약 |
| 최종 확정 | 후보, 응답, 조정 반영 결과 | `MeetingDraft.confirmed` | 확정 안내 |

## 14. 검증 필요 항목

| 조건 변경 제안 분리 | AI 추론 범위를 줄이고 주최자 책임을 명확히 함 | 주최자가 조정안과 조건 변경의 차이를 이해하는가 |
